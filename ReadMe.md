# 🥗 Calorey: Multi-Agent Weekly Nutritionist

Calorey is a stateful, multi-agent AI system built with LangGraph and Flask. It automates the complex task of weekly meal planning; empowering users to eat healthier diets and acheive their nutrition goals. Calorey simulates a professional nutrition team: an Orchestrator to handle the consultation, a Master Planner to design the menu, and Specialized Workers to research and validate every recipe.

[Live deployment of Calorey](https://nutrition-agent-285051111133.europe-west2.run.app)

## 🚩 The Problem

Many people have the sincere desire to eat healthier and reach specific fitness goals, but due to the time pressures of life they don't have the time to spend planning balanced and structured meals to help them achieve these goals. The problem can be broken down into 3 main areas:

- **Decision Fatigue**: After a long day, deciding what to eat that fits your macros is mentally exhausting.

- **The "Easy Meal" Trap**: Busy schedules lead people to resort to convenient, unbalanced meals that don't align with their weight loss or muscle gain goals.

- **Constraint Complexity**: Manually calculating calories and protein while ensuring the meal contains varied enough ingredients to contribute a healthy diet is complex and often discourages people.

This results in many people failing to achieve their goals affecting both their physical and mental health.

## 💡 The Solution: Calorey Agent

Calorey solves this by acting as a stateful nutrition assistant. It removes the mental load by:

- **Conversational Profiling**: Instead of a rigid form, it extracts your data through natural chat.

- **Automated Logic**: It handles the complex math of TDEE and macro-splits automatically.

- **Agentic Research**: It doesn't just "hallucinate" recipes; it uses Tavily to find real, verified meals from the web that fit your specific caloric and dietary needs.

When combined it allows Calorey to produce a custom meal plan intentionally designed to fit a users requirements and preferences, allowing them to acheive their nutrition goals and enjoy their food whilst doing it.

## 🧠 System Architecture

Calorey uses a Directed Acyclic Graph (DAG) built on LangGraph to manage state and complex logic. The system is fundamentally schema-driven, utilizing Pydantic to enforce structural integrity across the agentic workflow.

### Schema-First Design (Pydantic Models)
To ensure the LLM outputs "machine-readable" data, the system relies on strict Pydantic models:

- **State Management**: The AgentState and WorkerState use Annotated types and custom reducers (like reduce_plan) to merge meal data from multiple parallel workers into a single weekly view without overwriting previous results.

- **Structured Extraction**: The ProfileExtract model acts as a "filter," ensuring the agent only updates fields the user explicitly mentioned (e.g., age, weight) while ignoring conversational filler.

- **Nutritional Integrity**: The Meal and Ingredient models enforce type-safety for calories, proteins, and allergens. This allows the system to perform mathematical validation and scaling on the data before it ever reaches the user.

- **Planning Constraints**: The planner_node_output model forces the LLM to generate a specific DayConstraint for every day of the week, ensuring cuisine variety and protein diversity are "baked into" the graph's instructions.

### Profile Extraction & Persistence
Uses gpt-4o-mini to extract physical metrics (height, weight, etc.) and dietary restrictions from natural conversation, persisting them to Google Firestore. Weights are automatically converted to cm and kg by the LLM to work with the metric calculations

### Automated Macro Calculation
A dedicated MetricsCalculator automatically determines TDEE, protein targets, and caloric needs based on the Mifflin-St Jeor equation.

- **BMR** = (10 x weight) + (6.25 x height) - (5 x age) + (5 if male or -161 if female)
- **TDEE** = BMR * Activity level multiplier (1.2-1.9)
- **Calorie target** = TDEE + goal modifier (do you want to gain or lose weight)

- **Protein target** = weight x (1.6 if gaining muscle or 0.8 if maintaining/losing weight)

The Calorie and Protein targets underpin the meal plan generation alongside dietary requirements and disliked foods.

### Worker Agent
All meal generation is carried out by the same sub-agent. The sub-agent is itself a Stategraph with a unique worker state, such that each worker is only aware of their own work. The graph consists of 4 nodes:

- **Search Node**: Responsible for meal generation equipped with a Tavily search tool, the node produces the proposed meal taking into account the users profile and the constraints on that day (more on this later).

- **Format Node**: Takes the meal generated by the search node and outputs it in Calorey's custom Meal schema.

- **Validation Node**: Fully deterministic, no LLM, the node ensures that the suggested meal conforms with the users requirements. A conditional edge exits this node which will route to the search node any errors in the meal for regeneration, or to the package node if there are no errors.

- **Package Node**: This node is responsible for integrating the workers generated meal into the main agent state ensuring the meal is stored correctly for later extraction in the frontend display. This node signals the end of the worker graph.

### The Fan-Out Pattern
- **Master Planner**: Creates a high-level 7-day skeleton ensuring cuisine variety.

- **Fan-out pattern**: The system "fans out" to multiple instances of the worker agent simultaneously (Breakfast, Lunch, and 7 individual Dinner workers).

- **Fan-in**: Langgraph automatically waits for the individual workers to all finish before moving to the next node of the graph, this ensures that the state is fully updated with the generated meal plan before the plan is saved to firestore for display.

## Key Engineering decisions

### Fan-out pattern and meal skeleton
When first testing the system there were concerns over the duration of the meal plan generation. Even creating only one breakfast and lunch with a weeks worth of dinners results in the creation of 9 meals. Each meal generation takes ~30-40 seconds, therefore, sequential meal generation would have taken ~5 minutes. This raised concerns over latency and potential damage to user experience (UX).

To solve this the Langgraph `send()` function was looked at. The function allows the parallel invocation of the same sub-agent as many times as required. Known as the fan-out pattern, parallel meal generation would reduce latency to the duration of only the slowest single meal generation, acting as the slow step. The method also gracefully handled instances when test users didn't require a breakfast, as the graph logic could simply not send an invokation to the sub-agent for breakfast.

Parallel workers created a new problem. It was hypothesised that the 7 dinner workers would likely create the same or very similar meals, given they had identical instructions and profiles to work off whilst be oblivous to the work being done by the other workers. This presented a major problem as not many users would be happy eating the same meal for dinner everynight.

To combat this issue a meal skeleton would be created before the fan-out. The skeleton would include a cuisine theme for that night and a main protein source to ensure a diverse range of proteins. The skeleton was stored as a list of `DayConstraint` models, visible in `models.py` each day constraint would be added into the individual `send()` calls, such that an individual worker was only aware of their personal constraints.

The result was a weekly plan filled with diverse cusines and protein sources to offer a more enjoyable eating experience.
